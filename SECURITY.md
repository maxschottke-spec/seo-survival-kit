# Security Considerations

This document explains what the skills in this plugin do at runtime, what they access, and what trust decisions you make when installing them.

## What the skills do at runtime

### `post-core-update-recovery`

- **Zero code execution.** Pure markdown / framework skill.
- No file I/O beyond Claude reading the SKILL.md.
- Zero network calls.
- **Risk: minimal.**

### `seo-outreach-report`

Contains four Node.js scripts that run when the skill is invoked:

| Script | What it does | What it accesses |
|--------|--------------|-------------------|
| `seo-audit-fetch-v2.js` | HTTPS calls to Sistrix, DataForSEO, Google PSI | Network (outbound only), reads `.env`, writes `/tmp/seo-*.json` |
| `seo-extract-v2.js` | Pure JSON parsing | Reads `/tmp/seo-*-raw.json`, writes `/tmp/seo-*-summary.json` |
| `seo-onpage.js` | Regex parsing of local HTML files | Reads `/tmp/*-home.html`, writes `/tmp/seo-onpage.json` |
| `seo-report-gen.js` | Builds HTML, then launches `/Applications/Google Chrome.app/Contents/MacOS/Google Chrome --headless --print-to-pdf=...` | Writes HTML to `~/Downloads/seo-audits/`, launches Chrome subprocess, writes PDF to `~/Downloads/` |

## Threats and mitigations

### Credential handling
- API keys are loaded from a `.env` file. **The .env file is in `.gitignore` and must NEVER be committed.**
- Scripts use `process.env.*` access — keys are not logged, not written to disk by the scripts themselves.
- **Your responsibility:** keep the `.env` file in a private location (e.g. `~/.config/seo-rescue/.env`, mode 600).

### Subprocess execution (Chrome)
- The PDF render launches `Google Chrome.app` with `--headless --disable-gpu --print-to-pdf`. The Chrome sandbox is **not disabled** (we removed the `--no-sandbox` flag in v0.1.0 after the initial release).
- The HTML being rendered is local-only and generated by our scripts.
- The HTML does NOT execute any user-controlled JavaScript — only the static report template.
- **Slug validation:** target slugs from `audit-config.json` are validated with `safeSlug()` (regex `^[a-z0-9][a-z0-9_-]{0,63}$`) before being interpolated into the Chrome subprocess command. Shell metacharacters in slugs are rejected with an error.
- **Path quoting:** the output PDF path is serialized with `JSON.stringify()` before being passed to `execSync`, ensuring proper shell escaping for paths with spaces or special characters.
- **Chrome binary path:** loaded from `CHROME_PATH` env var (default: `/Applications/Google Chrome.app/...`). Validated to exist before subprocess launch.
- **Verify:** if you want to be sure, read `seo-report-gen.js` — the entire pipeline is one file, no minification, no obfuscation.

### Network access
Scripts make HTTPS calls only to these hosts:
- `api.sistrix.com` — VI data
- `api.dataforseo.com` — SERP/backlinks data
- `www.googleapis.com` — PageSpeed Insights
- Target domain (via `curl`/WebFetch in some flows) — to fetch the homepage for on-page analysis

No data is sent to the plugin author or any third party beyond these named APIs.

### File system writes
Scripts write to:
- `/tmp/seo-*.{json,html}` — intermediate working files (you can delete after run)
- `~/Downloads/SEO-Auswertung-<domain>-<date>.pdf` — final output
- `~/Downloads/seo-audits/<slug>/report.html` — intermediate HTML

No writes outside these paths.

### What the plugin does NOT do
- Does not collect telemetry
- Does not phone home to any server controlled by the author
- Does not modify your Claude Code config / settings
- Does not modify any project files
- Does not execute code from the audited target domain (HTML is parsed via regex, not executed)
- Does not require sudo/root

## Trust model

When you install a Claude Code plugin from any third-party marketplace, you are granting that plugin the ability to:
- Run code in your shell with your user privileges
- Read files Claude has access to
- Access the network

This applies to ALL plugins, not just this one. Anthropic's docs are explicit about this — see [Plugin security warning](https://code.claude.com/docs/en/discover-plugins#security).

**Mitigations:**
1. Read the source before installing. This repo is intentionally small (2 skills, 4 scripts, ~1000 lines total).
2. Pin to a specific git commit if you want reproducibility:
   ```
   /plugin marketplace add maxschottke-spec/seo-survival-kit#<commit-sha>
   ```
3. Use the `local` installation scope if you only need it in one repo.

## Reporting security issues

If you find a vulnerability, please open a GitHub issue (for non-sensitive items) or email the maintainer directly (for items that could affect users with the plugin already installed).

## Audit trail

This plugin is open-source under MIT. Every change is visible in git history. There are no minified scripts, no obfuscated code, no compiled binaries.
